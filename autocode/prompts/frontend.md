## Frontend (TypeScript / React) Conventions

You are implementing a frontend issue. In addition to the universal TDD workflow above, follow these rules.

### TypeScript
- **No `any`** anywhere. If a type is genuinely unknown at a boundary, use `unknown` and narrow before use.
- Prefer `type` for unions/intersections, `interface` for object shapes that may be extended.
- Strict null checks are on; do not silence them with `!` unless a comment explains why the value cannot be null at that point.

### React
- Functional components only. PascalCase. Props typed via an `interface <Component>Props`.
- One component per file. File name matches the component name.
- Side effects belong in custom hooks (`use<Name>`), not inline `useEffect` blocks scattered through components.
- No `useEffect` for data fetching — use **TanStack Query** (`useQuery`, `useMutation`).
- Memoise only when profiling shows it matters. Don't sprinkle `useMemo` / `useCallback` defensively.

### Data layer
- API access lives in `src/api/`. The contents of `src/api/` are **generated by Orval** from the OpenAPI spec — **never edit these files manually**. If the contract needs to change, change the backend OpenAPI definition and regenerate.
- Mocks are generated by **MSW** from the same spec; do not write manual `fetch` or `axios` mocks.

### Testing
- **Vitest** for unit/component tests, **React Testing Library** for component behaviour.
- Test what the user sees and does — query by role/text, not by `data-testid` unless absolutely necessary.
- Each test sets up its own MSW handlers; do not share mutable handler state across tests.

### Styling
- Use the existing design tokens / Tailwind utilities. Do not introduce a new styling system.
- No inline `style={{ ... }}` for anything that has a token or utility equivalent.

### Build
- After implementation, run `npm run typecheck` and `npm run test` (or the project equivalents) and ensure both pass.
